# Project Migration Plan: kcptun-android-jedi

This document outlines the phased migration of the `kcptun-android` project from a legacy Linux-based environment to a modern Windows-based development stack (JDK 17+, Kotlin 1.9+, Go 1.21+, and AGP 8.x).

---

## Phase 1: Environment Baseline & Toolchain Setup
**Goal:** Establish a functional Windows development environment and identify path discrepancies.

**Prompt:**
> "Analyze the local Windows environment to verify the installation of JDK (prefer 17 or 21), Go (1.21+), and the Android SDK/NDK. Update the `local.properties` file or environment variables to point to the correct Windows paths for `sdk.dir` and `ndk.dir`. Specifically, identify the LLVM toolchain path within the Windows NDK installation to replace the hardcoded Linux paths in the build scripts."

**Next Step:** Once the environment is validated, proceed to Phase 2 to modernize the build system core.

---

## Phase 2: Gradle & AGP Modernization
**Goal:** Upgrade the build engine to support modern Java versions and faster build features.

**Prompt:**
> "Upgrade the Gradle Wrapper to version 8.x (e.g., 8.4) and update the Android Gradle Plugin (AGP) in the root `build.gradle` to version 8.x (e.g., 8.1.1). Adjust the `buildscript` repositories to ensure compatibility with the latest Maven Central and Google requirements. Resolve any deprecated Gradle DSL usage in `app/build.gradle` introduced by the AGP 8.0 breaking changes (e.g., namespace declarations)."

**Next Step:** With the build engine updated, move to Phase 3 to modernize the application code and SDK targets.

---

## Phase 3: Kotlin & Android SDK Upgrade
**Goal:** Align the project with current Android 14 (API 34) standards and Kotlin 1.9+ features.

**Prompt:**
> "Update `kotlinVersion` to 1.9.20 or later. In `app/build.gradle`, increase `compileSdkVersion` and `targetSdkVersion` to 34. Update the Kotlin standard library and any relevant AndroidX dependencies to their latest stable versions compatible with Kotlin 1.9. Ensure that the `jvmTarget` is set to 17 or higher to match the new toolchain."

**Next Step:** After the Android wrapper is updated, tackle the most complex part: the Windows-native Go build system in Phase 4.

---

## Phase 4: Windows-Native Go Build System (Porting Bash to PowerShell)
**Goal:** Enable native Go compilation on Windows without requiring WSL.

**Prompt:**
> "Convert the `app/src/make.bash` script into a PowerShell script (`app/src/make.ps1`). The new script must:
> 1. Dynamically locate the NDK LLVM toolchain on Windows.
> 2. Set the appropriate environment variables (`GOOS=android`, `GOARCH`, `CGO_ENABLED=1`, `CC`).
> 3. Use the `.cmd` or `.exe` versions of the NDK clang compilers.
> 4. Implement the same cross-compilation logic for `armeabi-v7a`, `arm64-v8a`, `x86`, and `x86_64`.
> 5. Handle the 'strip' utility equivalent for Windows NDK."

**Next Step:** Integrate the new PowerShell script into the Gradle lifecycle in Phase 5.

---

## Phase 5: Build Lifecycle Integration
**Goal:** Re-enable the native build tasks within the Windows Gradle execution.

**Prompt:**
> "Modify the `goBuild` and `goClean` tasks in `app/build.gradle`. Remove the Windows restriction and update the tasks to execute `powershell.exe -ExecutionPolicy Bypass -File src/make.ps1` when running on Windows. Ensure the `jniLibs` source sets correctly pick up the `.so` files generated by the new PowerShell script."

**Next Step:** Perform final verification and cleanup in Phase 6.

---

## Phase 6: Validation, Testing & Documentation
**Goal:** Ensure technical integrity and update project documentation for future Windows developers.

**Prompt:**
> "Perform a clean build using `./gradlew assembleRelease`. Verify that the generated APK contains the correct native libraries for all ABIs using `analyze-apk` or manual inspection. Update `GEMINI.md` and `README.md` to reflect the new Windows-compatible build instructions and toolchain requirements. Run the existing test suite to ensure no regressions were introduced during the migration."

**Final State:** The project is fully modernized and buildable on Windows.
